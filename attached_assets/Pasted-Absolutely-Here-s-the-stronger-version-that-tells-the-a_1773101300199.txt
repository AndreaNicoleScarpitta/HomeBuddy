Slug Standards

Page Slugs

Use stable, human-readable kebab-case slugs.

Examples:

dashboard
task-list
system-details
document-vault
profile-settings
maintenance-calendar
file-upload

Modal Slugs

Use the prefix modal- followed by a kebab-case purpose.

Examples:

modal-add-system
modal-edit-system
modal-add-task
modal-edit-task
modal-delete-confirm
modal-document-analysis
modal-file-upload
modal-donation-prompt
modal-settings

Format Rules

All slugs must:

be lowercase

use only letters, numbers, and hyphens

contain no spaces or underscores

be stable over time

be globally unique across both pages and modals


Regex:

^[a-z0-9]+(-[a-z0-9]+)*$

Note: dashboard and modal-dashboard are distinct and valid, but two components may not share the exact same slug.

Auto-Generation Rules

If an existing page/modal already has a slug-like identifier, normalize and preserve intent where possible.

If a page/modal has no slug:

1. derive it from:

route path

file name

component name

feature context



2. convert to kebab-case


3. ensure uniqueness


4. if collision occurs, append a stable disambiguator based on function or route context



Examples:

ProfilePage.tsx → profile

DocumentAnalysisModal.tsx → modal-document-analysis

SettingsDialog.tsx → modal-settings

TaskEditSheet.tsx → modal-edit-task


If two pages would both become settings, disambiguate logically:

Do not use random IDs in slugs unless there is absolutely no stable disambiguator.

Central Registry

Create a single central registry, for example:

src/lib/slugRegistry.ts

Structure example:

export const PAGE_SLUGS = {
  dashboard: "dashboard",
  systems: "systems",
  systemDetails: "system-details",
  tasks: "task-list",
  documentVault: "document-vault",
  profile: "profile",
};

export const MODAL_SLUGS = {
  addSystem: "modal-add-system",
  editSystem: "modal-edit-system",
  addTask: "modal-add-task",
  editTask: "modal-edit-task",
  deleteConfirm: "modal-delete-confirm",
  fileUpload: "modal-file-upload",
  documentAnalysis: "modal-document-analysis",
};

export const ALL_SLUGS = [
  ...Object.values(PAGE_SLUGS),
  ...Object.values(MODAL_SLUGS),
];

Every page and modal should reference this registry rather than duplicating strings inline.

Component Integration

Pages

Every page component must expose or consume its slug from the registry.

Acceptable patterns:

const PAGE_SLUG = PAGE_SLUGS.dashboard;

or

export const pageSlug = PAGE_SLUGS.dashboard;

On page load or mount, the page should trigger tracking:

trackPageView(PAGE_SLUG);

Modals

Every modal component must expose or consume its slug from the registry.

const MODAL_SLUG = MODAL_SLUGS.addTask;

When the modal opens, trigger:

trackModalView(MODAL_SLUG);

Only fire modal tracking when the modal becomes visible/open, not merely when imported.

Validation Utility

Create a runtime validation utility that checks slug quality and uniqueness during app startup in development and test environments.

Example:

export function validateUniqueSlugs(slugs: string[]) {
  const seen = new Set<string>();

  for (const slug of slugs) {
    if (!slug) {
      throw new Error("Empty slug detected");
    }

    if (!/^[a-z0-9]+(-[a-z0-9]+)*$/.test(slug)) {
      throw new Error(`Invalid slug format: ${slug}`);
    }

    if (seen.has(slug)) {
      throw new Error(`Duplicate slug detected: ${slug}`);
    }

    seen.add(slug);
  }
}

Invoke this against the registry.

Tests Required

Add comprehensive tests.

1. Registry Uniqueness Test

Ensure all page and modal slugs are globally unique.

expect(ALL_SLUGS.length).toBe(new Set(ALL_SLUGS).size);

2. Format Test

Ensure all slugs match the kebab-case regex.

ALL_SLUGS.forEach(slug => {
  expect(slug).toMatch(/^[a-z0-9]+(-[a-z0-9]+)*$/);
});

3. Coverage Test for Pages

Automatically verify all discovered pages are represented in the slug registry.

This can be done by:

comparing route/page files against the registry

or requiring each discovered page component to export or consume a slug


Fail if a discovered page lacks a mapped slug.

4. Coverage Test for Modals

Automatically verify all discovered modals are represented in the slug registry.

Fail if:

a modal/dialog/sheet/drawer component exists without a slug

or a visible workflow surface is missing tracking identity


5. Page Tracking Test

Confirm page tracking fires with the correct slug on page render or focus.

6. Modal Tracking Test

Confirm modal tracking fires when a modal actually opens.

7. Duplicate Failure Test

Intentionally validate that duplicate slugs throw the expected error.

Naming Heuristics

Use these naming heuristics when generating slugs:

For Pages

Priority order:

1. route path

3. file name


4. component name


5. folder context

For Modals

Priority order:

1. user action purpose


2. file/component name


3. parent feature context
